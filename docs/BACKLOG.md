### Native wrapper / Syphon — does going native FORK the code? (analysis, 2026-06-10)
Daniel's open question + the answer, because it shapes everything:
- **NO fork, if we use a WRAPPER.** Electron (macOS) and Capacitor/WKWebView (iOS/iPadOS/macOS, App Store) both RUN the existing web app — they reuse the Engine/Kit/Components/Chrome as-is. "Native" = a thin shell + native modules (Syphon, camera, HDMI). Every UI/IxD/engine refinement we make here carries over for free — **no duplication.** Only a full SwiftUI+Metal REWRITE would fork, and that is NOT required for Syphon/camera. So: keep this single web codebase as the source of truth; native is a shell. This de-risks Daniel's worry — polishing now is not wasted.
- **Stay here vs Xcode:** Electron is Node-based — no Xcode, native modules via node-gyp, and **no Apple dev license needed to run/distribute locally** → ideal for Daniel's "validate Syphon at shows first, decide how to sell later." Capacitor (for the eventual iOS/macOS App Store) generates an Xcode project that BUILDS the web app + shell; you still DEVELOP here and sync, with Xcode only for native-plugin code + signing.
- **The real technical unknown = getting the WebGL output INTO Syphon efficiently.** Syphon shares GPU textures (IOSurface/Metal). The easy path (canvas `readPixels` → CPU → Syphon) may be too slow for live high-res output (same readback/`VideoFrame` bottleneck as export); the fast path (shared texture / Electron offscreen GPU interop) is the risk. → **A small Electron + Syphon SPIKE** answers it: prove the frame-out path + measure latency at performance resolution, plus a native hook (live-feed → full-res still), before building features. No Apple license to start.
- **SPIKE COMPLETE + END-TO-END CONFIRMED (2026-06-17, `spike/electron-syphon/`).** CPU readback path (`drawImage` + `getImageData`) is viable — no GPU texture sharing needed. Tested on M5 Max MacBook Pro running Electron 24 / Chromium / Blink. Results: **960×960 → ~120fps / 2.3ms capture; 1920×1920 → ~95fps / 8.5ms capture; 3840×2160 (16:9 4K) → ~41fps / 18.9ms capture; 3840×3840 → ~23fps — not viable.** Bottleneck at all resolutions is GPU→CPU pixel readback (`getImageData`), not IPC. At 1920×1920 the headroom is large enough to absorb Arena competing for GPU. At 4K 16:9 there's less margin — viable on clean hardware, may dip below 30fps under full system load. **Hardware note:** unified memory (Apple Silicon, M1+) is the key enabler — no PCIe bus crossing for readback. Viable on M1+ at 1920×1920; 4K more suited to M1 Pro/Max and above. Intel Mac likely not viable. **Canvas sizing gotcha:** app canvas stayed at 300×150 in Electron (CSS-scaled display, width/height attrs not set by app's resize logic). Real integration should drive Syphon from the engine's FBO export path (`renderToFBO`), not the display canvas — this gives a resolution-independent output path and avoids the sizing issue entirely. **Syphon native addon proven:** used `node-syphon@1.5.0` (`SyphonMetalServer`) — no custom ObjC++ needed. `SyphonOpenGLServer` failed in Electron (GPU process isolation); `SyphonMetalServer` works in the main process on Apple Silicon. Resolume Arena confirmed "Electron - Fold" as a live Syphon source. Spike code in `spike/electron-syphon/` seeds the real `host.syphon` implementation.
- **Recommendation:** think about the wrapper workflow SOONER (capability-gate native-touching features; avoid web-only assumptions) even though it doesn't block current web work; do the Electron-Syphon spike as the technical experiment that de-risks the whole native/Syphon path (and Daniel's personal live-performance milestone). Sequence: engine hardening (now) → realtime-video-to-disk (web) → Electron-Syphon spike → native features.
- **Architectural prep for the wrapper — IN PROGRESS (plan `~/.claude/plans/we-re-doing-architectural-preparation-indexed-peach.md`).** Two behavior-preserving refactors so a third entry point (Electron/Capacitor) mounts the same codebase without forking `main.js`: (1) split the app *wiring* from the desktop *chrome*; (2) add a runtime-capability layer — a **capability-profile** module (centralize the scattered per-engine decisions: capture path, codec ceiling, FBO/texture caps, readback strategy, seek/orientation/ProRes — into one queryable table across the 5 runtimes) and a **host-services** interface (Syphon/MIDI/native-camera/file-system seam with a web no-op `webHost`, so the app queries + degrades and native shells later mount against a stable interface). **Phase 1 SHIPPED (Build 164):** closure→env migration — every module-level mutable `let` the wiring closed over now lives on cohesive `env` runtime sub-objects (`media`/`live`/`motionRT`/`clip`/`filmstrip`/`scrub`/`sched`/`sourcePreview`); no behavior change. **Phase 2a SHIPPED (Build 165):** clip editor extracted to `shell/clip-editor.js` (`createClipEditor(env)`); established the cross-module convention — late-bound `env.X()` handles between modules, public surface hung back on `env`, stable refs (`engine`/`motion`/`session`) destructured. **Phase 2b SHIPPED (Build 166):** source host extracted to `shell/source-host.js` (`createSourceHost(env)` — media load + live camera + still export). **Phase 2c SHIPPED (Build 167):** motion runtime extracted to `shell/motion-runtime.js` (`createMotionRuntime(env)` — sampling/playback/keyframes/timeline/filmstrip/scrub/retime/motion-JSON/video-export sheet, ~1,100 lines, byte-exact extraction). **main.js is now the desktop chrome: 2636→865 lines.** **Phase 2d SHIPPED (Build 168) — PHASE 2 COMPLETE:** `shell/app.js` `createApp(env, { host, capabilities })` mounts the wiring (clip → source → motion) in one call + threads the injectable host/capability seams. **Phase 3 SHIPPED (Build 169):** `kit/capabilities.js` `createCapabilities(engine)` → probe-once table (engineId / capturePath / maxFBOSize / maxTextureSize / firefoxTextureCapped), threaded onto `env.capabilities`; ## next up — new forms

These all benefit equally from the registry architecture — each is one new file in `src/engine/forms/` plus one line in the registry. Order is rough; pick whichever sounds most fun.

- **Hyperbolic Escher (circle limit).** Tessellation of the Poincaré disk model of hyperbolic geometry. Output is a circular image with shapes crowding toward the edge, like Escher's *Circle Limit* prints. Heavy lift: needs custom overlay (circular disk boundary + warped fundamental triangle) and custom controls (Schläfli tiling selector). The Droste form (Build 41) introduced the `drawOverlay` / `classifyPointer` schema hooks that this form can reuse — the engine-schema-extension lift is already done. Distinctive Escher-feel; significant aesthetic differentiation.
- **p31m wallpaper (future).** Alternate triangular tiling — same equilateral triangles as p3m1, but with mirror axes running through vertices rather than along edges. Fully seamless (passes the no-visible-seams design constraint). Visually distinct from p3m1, especially at triangle centers vs. corners. Lower priority than the above; vocabulary expansion rather than a foundational form.
- **Radial polygon-frame variation (low priority).** Cosmetic enhancement to the existing radial form: optionally render with an n-sided polygon outer boundary instead of a circular arc. Same fold math, same n-fold rotational symmetry, just a different visible frame shape. Constrained to even sides matching segment count (4-segment radial → square frame, 6 → hex frame, 8 → octagon frame, etc.) for seam compliance. May emerge organically as a side effect of tile-aware features since polygon framing relates to tileable output shapes. Not a separate form; a parameter on radial.

**Design constraint for all new forms:** No visible seams on any output. Forms without sufficient mirror symmetry (pinwheel-only patterns like p3, p6, p4, etc.) are explicitly excluded — they show seams between fold cells which breaks the kaleidoscope illusion. Glide-reflection groups (pmg, pgg) are also excluded because glide axes can produce visible discontinuities depending on source image content. Rectangular mirror groups (pmm, cmm) are excluded for a different reason — they're visually redundant with the existing square form's aspect-ratio control. With p3m1 shipped (Build 32), p31m is the only remaining wallpaper group that adds distinct visual vocabulary while reliably satisfying the seam constraint.

Pairs well with the live-camera shell. Having more forms available makes the camera shell more demoable and surfaces form-switching UX issues earlier. New forms can drop in at any time as opportunistic parallel work; they cannot destabilize other phases because each form is a self-contained file plus a registry entry.

For each new form, also fill in `tilesPerDim(state)` so the resolution hint is accurate.

## next up — capability tier

**Layering vocabulary (settled — refined to Engine / Kit / Components / Chrome).** "Shell" was conflating layers with different reuse profiles, which made each use case sound like a rebuild. It isn't:

- **Engine** (`src/engine/`) — forms, shader, gl, geometry. Pure pixels, zero DOM. Never rebuilt.
- **Kit** (`src/kit/`, `shell/state.js`, `history.js`, `params.js`) — DOM-agnostic primitives shared by every front-end: state schema, undo/redo, **snaps** (droste arms/spiral), parameter registry, tween/easing + keyframe model (to build). Plus **Host services** (composable, not UI): camera (`shell/camera.js`), render-driver loop, export (`engine.exportAt` + `shell/zip.js`). Grows over time; never rebuilt.
- **Components** (`src/components/`) — mountable UI mounted by BOTH chromes, **parameterized (e.g. touch-target size), not forked**: the **source-overlay** (draw + hit-test + the expensive proportionality/mirroring gesture math), **output gestures** (pinch/twist), and the **param-control renderer**. The source-overlay's event handling must never be reimplemented per chrome. Never rebuilt.
- **Chrome** (`src/desktop/`, `src/mobile/`) — layout, divider, tab bar, disclosure, gesture routing, mode detection. The only layer genuinely rebuilt per device.

**Two front-ends, plus a deferred third:** the **desktop chrome** (current, extended — serves desktop still, still animation, video animation, and live camera on **iPad**, which stays on the desktop chrome) and the **mobile chrome** (new — touch, phone-class viewports; serves mobile still editor, mobile camera, live capture-to-disk). ### Daniel's initial sketch

### Counter-perspective (from prior conversation, captured for discussion)

- The 4-step flow describes the *photo-import* path. The *camera-first* path probably wants a different entry: camera is already live, kaleidoscope is already on screen, the interaction is "frame the world, capture." Camera live as the default mode on mobile.
- On split-screen wedge-and-preview: consider wedge overlay drawn *on top of the live camera feed* rather than in a split. Phone is small; every pixel counts. Toggle between "wedge view" (camera + overlay, no kaleidoscope) and "kaleidoscope view" (full-screen output) with a single tap.
- On hamburger vs. tab bar: argued for tab bar because discoverability matters when showing this to friends who've never seen it; hamburger hides everything behind one tap. Counter-argument: text labels matter and tab bar may not have room.

### node-syphon publishImageData memory leak — RESOLVED (Build 185), PR'd upstream as [#46](https://github.com/benoitlahoz/node-syphon/pull/46)

**TL;DR (the rest of this item is historical investigation — skip unless reviving):** their native `publishImageData` allocated a Metal texture per frame and never released it (~14.7MB/frame; their own commented-out TODO). Fix = a LOCAL texture released in `addCompletedHandler` (~5 lines), built with node-gyp + Command Line Tools (N-API, no full Xcode), carried as a vendored binary `electron/vendor/node-syphon/syphon.node` applied by the postinstall hook `electron/scripts/patch-node-syphon.cjs`. Profiler-flat + Arena-confirmed. **REVERT when #46 merges + releases:** bump `node-syphon` in `electron/package.json`, delete the hook line + `electron/vendor/`. node-syphon is GPL-3.0+ (compatible with Fold's AGPL). Deferred: a universal (x86_64+arm64) binary at packaging time (arm64-only now).


**ROOT-CAUSED (Build 183).** node-syphon@1.5.0's NATIVE `SyphonMetalServer.publishImageData` leaks **~14.2 MB per call** (exactly one 1920² RGBA frame) and never frees it. Proof: a headless probe (fixed reused buffer, `node --expose-gc`, loop publishImageData) showed RSS climbing linearly ~14.2MB/call while V8 `heapUsed`/`external`/`arrayBuffers` stayed flat and both `gc()` and `dispose()` reclaimed nothing → it's native (Metal texture/IOSurface), invisible to V8. NOT our code: the JS wrapper (`dist/node-syphon.js`) is a pure pass-through (no copy/retain), our IPC handler is fire-and-return, and the Build-182 backpressure fix already bounded the IPC queue (it turned a ~100GB/min runaway into a paced ~14MB-per-published-frame climb). **1.5.0 is the latest published version** (checked `npm view`), so no upgrade fixes it. At FHD/1080p (~8MB/frame) and ~12fps the climb is ~100MB/s; at 1920² (~14.7MB) higher — minutes to OOM either way. **Options:** **(b) periodic server recycle — RULED OUT (Daniel device test, Build 184):** disarming broadcast halts growth but memory does NOT recover (3.71→3.68 GB, no further drop over minutes); only quitting Electron frees it. So `dispose()` does not reclaim the leaked native memory even on a live run loop — recycling the server wouldn't help. **(c) lower output resolution — REJECTED by Daniel** (cuts the per-frame leak proportionally but only buys time; not acceptable). **Remaining, both require native macOS work:** **(a) IOSurface-reuse publish** — allocate ONE IOSurface, write pixels into it each frame, publish via `publishSurfaceHandle` (which node-syphon already exposes); Syphon is built around IOSurface sharing, so this is the architecturally-correct zero-leak AND near-zero-copy path (also fixes the throughput/fps item). Needs a small native addon (or a lib) to create/write an IOSurface from JS — the first native code we'd own (build toolchain, N-API for ABI stability, code-signing/notarization when packaging — ties to the deferred electron-builder). Daniel leans here ("architecturally correct"). **(d) fork + patch node-syphon's native source** — the leak is in their `MetalServer.publishImageData` (the JS wrapper is pass-through); patching it to release/reuse the texture reuses their existing Metal plumbing (likely a smaller change than (a)), at the cost of building node-syphon from source + maintaining a fork (or landing a PR upstream — 1.5.0 is latest, so check if the maintainer is active). **INVESTIGATION DONE (2026-06-18 — package dig, alternatives, "why us"):**
- **Known + unfixed upstream.** `benoitlahoz/node-syphon` issue [#45 "Memory Leak"](https://github.com/benoitlahoz/node-syphon/issues/45) is OPEN (another user, Felix Turner, same symptom — he attributed it to Electron IPC, but our headless probe proves the deeper leak is in native `publishImageData` itself, IPC-independent). Closed #12 was an earlier Electron leak. So it's a real node-syphon bug, not our code.
- **Why it hits US hard:** Syphon is DESIGNED for servers to publish a GPU texture (`publishFrameTexture:`, data stays on the GPU — confirmed in Syphon-Framework docs). We use the off-path `publishImageData` (raw CPU pixels) because our frames come back from a WebGL readback as CPU bytes. Most Syphon apps render on the GPU and publish textures, so they never exercise the leaky CPU path. We push full-res CPU frames continuously → we hit it at full tilt.
- **A NON-LEAKING reference exists:** `vcync/electron-syphon` by 2xAA (author of the modV VJ tool). In the #12 thread he states his CPU-pixel implementation "doesn't leak." It's a reference Electron app (a node-gyp native module + bundled `Syphon.framework`), NOT an npm package, so adopting it means pulling its native source into our `electron/` + a node-gyp build step. But it de-risks the native path enormously (proven non-leaking code for our exact case), and the author wanted collaborators.
- **Alternative TRANSPORT — NDI:** `Streampunk/grandiose` (Node NDI bindings, cross-platform Win/Mac/Linux). Resolume ingests NDI natively (always-on input). This sidesteps Syphon (and its leak) entirely and is cross-platform — strategically interesting for the planned motion/live shells beyond Mac. Tradeoffs: NDI is network/compressed (latency + quality vs Syphon's local GPU share), and frames still need to leave the renderer (same readback). Bigger architectural change.

**Revised options (was a/b/c/d):** **(A) Adopt the non-leaking native publisher** (extract/adapt `vcync/electron-syphon`'s module, or collaborate). Native build, but proven. **(B) NDI via grandiose** — different transport, cross-platform, no Syphon. **The GPU-texture-handle path** (zero readback, the real long-term answer + fixes fps) is what node-syphon's maintainer is "on the path of understanding" but it's unbuilt even upstream. **(C/our-own & D/fork-patch)** are now dominated by (A) (a proven reference beats writing fresh or patching the leaky one). **This is a Syphon-vs-NDI architecture fork — Daniel's call** (Syphon = lowest latency, Mac-only, what Arena prefers; NDI = cross-platform, fits the 3-app future, more overhead).
**Native build, plainly:** a C++/Objective-C `.node` addon compiled with **node-gyp** + Apple's **free command-line tools** (`xcode-select --install`; full Xcode IDE not required, code written in our normal editor, build is a CLI step). **No paid Apple Developer account for local/personal use** — that's only for distributing a signed/notarized app to other machines (ties to the deferred electron-builder). Runs IN-process (one app), not multiple apps.
**Stopgaps (no native, if we want safe experimentation meanwhile):** (i) a **safety auto-stop** — DROPPED (Daniel: less useful than the recycle); (ii) **helper-process recycle** — run the Syphon publisher in a separate child/utility process and kill+respawn it before OOM (process death DOES reclaim native memory, unlike `dispose()`); costs a brief Arena dropout per recycle + an extra IPC hop. Recycle-at-the-process-level works where recycle-at-the-object-level (option b) didn't. **VIABILITY UP (Daniel, 2026-06-18):** Arena auto-resumes playback from the named server near-instantly (he tested stop/restart, no noticeable delay), so the per-recycle flicker is brief. Cadence is the catch: at full res the leak is ~200-300MB/s, so a safe ceiling means recycling ~once/min, not every 2-3 min. Verdict: a viable FALLBACK / safety net, but the real fix is preferred.

### Diagnostics system — close the capability↔findings loop (spec, Claude 2026-06-17)

**STATUS — folded into Fold Live Phase 0 (the output-status chrome + the Chromium-perf question are both consumers of it).** Layer 2 (capability line in the readout) **landed in Build 171** — the diagnostics readout now prints `profile: engine · capture · maxFBO` beside the raw caps (relocated into `#diagSheet`). **The op-perf ring buffer landed in Build 175** (Increment 2): `kit/op-ring.js` `createOpRing(120)` → `env.diag.ops`, the output bus pushes the `op:'live-output'` record shape (per ~1s window: throughput + per-frame render/readback/publish ms), and the diagnostics sheet's `#diagOps` shows the recent records. **Still open here (NOT yet done):** (a) the **session report** — wiring `env.diag.ops` + `env.capabilities` into the "copy diagnostics" JSON (`gatherDiagnostics`/`showDiagnosticPanel` still emit only the static profile + webgl/probe/e2e blob; the op ring shows in the sheet but is not yet in the copy-JSON); (b) the **`still-export` / `video-render` / `clip-bake` op records** (only `live-output` emits the shape today — the three heavy ops still only print ephemeral status lines); (c) the clip-bake user-feedback line + the full cross-engine checklist.

**Problem.** Diagnostics today are real but scattered and ephemeral: (1) the always-on settings-panel readout shows RAW WebGL caps (renderer / max texture / max export / DPR) but NOT the DERIVED decisions the app actually makes — `engineId`, `capturePath` (gl vs 2d), `firefoxTextureCapped` — so a bug report can't say "gecko / 2d / 8K-capped"; (2) the rich "run diagnostics" JSON (`shell/diagnostics.js`) captures caps + FBO probe + e2e test but predates `kit/capabilities.js`, so it omits the capability profile; (3) per-op perf (the post-render line "rendered in 25.1s · 21 frames/s · /frame gl 40 · vframe 7 · encode 0 ms") is shown for **video render** and partially for **still export**, but is **ephemeral** (gone on the next action) and **absent for the clip bake**; (4) on-device findings ("Firefox felt stuttery") lose their runtime context between sessions, so we re-investigate. Goal: make every finding self-describing and paste-ready, cheaply and unobtrusively, while giving users light, friendly feedback.

**Design — three layers, separated by audience (don't conflate):**

1. **User feedback (light, always-on, friendly).** Keep the current post-op success lines but trim to the human-meaningful part ("saved ✓ · 17.5s @ 30fps · 25s to render"). No raw ms in the user's face. This layer reassures; it doesn't diagnose. **Add the missing one: a clip-bake progress/result line** (bake has none today — Daniel's note). Capability-driven *contextual hints* live here too, but earned + rare (we already do the Firefox-8K notice; a "this engine uses the slower 2D export path — Safari is ~Nx faster" hint could show only in the video-export sheet on non-WebKit, dismissible — not a nag).

2. **Diagnostics readout (always-on, cheap, dev-facing but visible).** Extend the settings-panel readout to print the **capability profile** beside the raw caps: `engine: gecko · capture: 2d · maxFBO 8192 · FF-capped`. One line, derived from `env.capabilities` (already computed — near-zero cost). This is the cheapest, highest-value piece (the "close the loop" item): it turns the panel into a glanceable answer to "what runtime decisions am I on right now?"

3. **Session report (opt-in, the paste-ready artifact).** A single **"copy diagnostics"** action (clipboard + optional download) that emits ONE JSON blob = the static profile (build, UA, platform, `env.capabilities`, the existing webgl/probe/e2e blob from `shell/diagnostics.js`) **+ a small ring buffer of the last N op-perf records**. This is what kills cross-session re-investigation: hit a stutter → copy → paste into the relevant BACKLOG finding, already carrying engineId/capturePath/the last render's per-frame timing.

**The connective tissue: a unified op-perf record.** Instrument the three heavy ops (still export, video render, clip bake) to emit the SAME shape and push it to a session ring buffer on `env` (e.g. `env.diag.ops` — ephemeral runtime state, fits the env-subobject pattern):
```
{ op:'video-render'|'still-export'|'clip-bake', w, h, frames, totalMs,
  perFrameMs:{ gl, vframe, encode }, throughputFps, engineId, capturePath, codec }
```
Video render already computes most of this (just route it to the buffer instead of only the status line); still export emits a subset; the clip bake emits nothing → add it there. Then both the user line (layer 1) and the session report (layer 3) read from the same record — no duplicate timing code.

**Principles:** cheap (capabilities is already computed; perf numbers already measured — this is mostly *routing + surfacing* existing data, not new measurement); unobtrusive (users see only the light line + an unintrusive readout; the heavy report is opt-in behind a button); single-source (profile from `kit/capabilities.js`, perf from the shared record — no scattered re-derivation). **Pairs with:** the cross-engine test checklist (a finding template seeded by the session report), the BACKLOG engine matrix (paste target), and the Chromium cross-browser pass above. **Sequence:** do it as part of / just before the stability+perf cycle, since that's when the loop pays off most. Smallest first valuable slice = layer 2 (capability line in the readout) + adding `env.capabilities` to the existing "run diagnostics" JSON.

## open architecture questions

Three former questions here are resolved by the layering vocabulary now recorded in the capability tier above (Engine / Kit / Host / chrome) — kept as brief settled notes so the reasoning isn't lost:

- **Engine input contract (→ Phase 0 spike).** The engine should accept HTMLImageElement, HTMLVideoElement, HTMLCanvasElement, ImageBitmap, and VideoFrame as a texture source, since `gl.texImage2D` natively accepts all of these. Confirm with the 30-min spike in Phase 0 before camera/video work; if it doesn't, the refactor is small.
- **Mobile is a distinct chrome (settled).** Not a responsive retrofit of the desktop chrome — a separate front-end pointed at the same engine, rendering the shared parameter registry. This is the commitment that makes the pro-and-playful product story possible (see `FOLD.md`).
- **Shared infrastructure for video sources (settled).** Camera (MediaStream), video file (`<video>.src = file`), and animated still (parameter timeline) are *host modules* over a common continuous render driver, not three separate code paths. The Phase 0.5 camera host is the first; video-file input (Phase 4) and the tween-driven timeline (Phase 3) plug into the same driver.
- **WebCodecs availability for video export (Phase 3 detail).** Prefer WebCodecs `VideoEncoder` for frame-perfect output; fall back to `MediaRecorder` if unsupported. Codec preference: mp4/h264 if available, webm/vp9 otherwise. May need to expose codec choice in advanced export settings.
